## Phase 5: Armature and Rigging

### What got built

**Armature structure:**
- Spine chain: root, spine.001, spine.002, spine.003, spine.004, neck, head
- Tail chain: tail.001, tail.002, tail.003, tail.004 (separate object, still bone-driven)
- Front legs: front_thigh.L/R, upper_front_leg.L/R, lower_front_leg.L/R
- Rear legs: rear_thigh.L/R, upper_back_leg.L/R, lower_back_leg.L/R

**Bone naming convention locked in:**
- Lowercase with underscores for body parts
- Dot notation for spine/tail numbering (spine.001)
- .L / .R suffix for sided bones (allows Blender's Symmetrize to work properly)

This naming becomes the standard for all future quadruped PJCC characters (Argus, Oskar) so a single rig template applies cleanly.

### Workflow used

1. Built left-side leg bones only (one front, one back)
2. Symmetrize to auto-generate right-side counterparts with correct parenting
3. Parented legs to spine bones with Keep Offset (not Connected — allows leg bones to move independently while following spine)
4. Parented each mesh object to armature with Automatic Weights:
   - Princess_Head (head + ears, merged object)
   - Princess_Body (body + legs, merged object)
   - Princess_Tail (separate object)
5. Tested in Pose Mode — deformation works on all bones
6. Cleared pose, set armature to rest position for master file

### Issues encountered and resolved

- **Stray "Bone.012" in outliner** — auto-named bone that didn't get renamed during build. Tracked down in edit mode and renamed. Vertex group on mesh still references it, harmless.
- **Inconsistent bone naming initially** — mixed camelCase, underscores, and different conventions. Cleaned up to lowercase+underscores+dot numbering before documenting.
- **Outliner restructuring confusion** — renaming armature object caused mesh objects to appear nested under it in outliner. Confirmed functional, just a visual change.

### Known acceptable v01 imperfections

- One knee loop cut ended up slightly diagonal — ignorable for v01
- More spine and tail bones than originally planned (extras provide more control)
- Weight paint on some verts is imperfect — minor deformation weirdness at mesh seams is acceptable for v01

**Test scene:** verified linked library workflow by:
1. Creating a fresh scene file
2. File → Link → selecting Princess collection from master
3. Making library override on the armature for pose control
4. Successfully posing Princess in the scene
5. Confirmed scene-specific pose does not affect master file

### Architectural decisions

**Approach: Edit-in-place with dated backups**
- Princess.blend is the single active master; scene files always reference this exact path
- Before any major change (future v02 build), copy to `_backups/Princess_vXX_backup_YYYY-MM-DD.blend`
- This provides seamless scene updates (zero per-scene work) + rollback safety
- No renaming of master file across versions — file path stability is the contract

**Why not separate v01/v02 files:**
Maintaining separate files per version would require updating every scene's link target on each upgrade. Given the scale of PJCC (many scenes planned), this would accumulate real overhead. Edit-in-place with backups solves both deployment speed and rollback needs.

### Swap-ready contract established

Created SWAP_CONTRACT.md documenting:
- Locked file path
- Locked collection name (Princess)
- Locked bone names
- Safe-to-change vs. will-break-scenes lists
- v02 swap procedure
- Backup protocol

This documentation protects against future-self accidentally breaking the linked library architecture by renaming something critical.

## Decisions made across the full build

**On breed accuracy:**
Princess is a pitbull/cattle dog mix in real life, but v01 prioritizes "reads as a dog" over "reads as specific breed." Silhouette and proportions are hound-leaning due to reference photos. v02 target can revisit proportions.

**On ears:**
Standing ears chosen despite real Princess having drop ears that often stand when alert. Acceptable for v01. Drop ears are a candidate v02 improvement.

**On topology:**
Local subdivides for leg attachment instead of global subdivide — keeps rest of mesh clean low-poly. Some topology is imperfect (angled loop cut, n-gons near leg bases) but rigging works regardless.

**On separate objects:**
Head+ears, body+legs, and tail kept as three objects instead of joining into one mesh. Simpler to edit, rigging still works via multi-object parenting to single armature. Small visible seams at joint boundaries accepted as v01 aesthetic.

**On workflow:**
Mirror Modifier workflow enforced throughout. Applied after final mesh was symmetric to reduce file complexity. All modifiers now applied on master file.
